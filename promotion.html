Project Hydra Labs: Promotion Process

Guidelines for Promotion to Project Hydra from Labs

  1. Version 1.0

  2. Good unit test coverage measured by community (e.g. 100% or 75% of what’s important)

    1. uses travis

    2. uses coveralls or simplecov

  3. Documentation

    1. README

      1. Statement of purpose

      2. Explicitly states whether there is a dependency on Hydra

      3. Lists all third-party dependencies

      4. Basic install

      5. List experimental features

    2. How to contribute -> CONTRIBUTING.md

    3. How/Who to contact for help -> push out to all gems like CONTRIBUTING.md

    4. Known Issues documented on issues tab

    5. Tutorial / walkthrough

  4. Must have any TODO items in documents addressed and removed

  5. Community use by three or more institutions

  6. In active use for six months

  7. Show compatibility with current Rails versions and other dependencies, when was it last tested; note compatibility with prior versions when available

    1. Compatability can be specified in the gemspec

  8. All Contributors should have Hydra CLA - Apache compatible license [Note for steering committee: what about non-Hydra contributions for gems that don’t require Hydra?]

  9. Hierarchy of promises asserted in clearly defined acceptance tests

 

Mechanism for Promotion

  1. At each Hydra partners meeting, gems are reviewed for promotion / graveyard

    1. Items up for promotion will have a breakout group for code review

    2. Group of contributors commit to upgrade path

    3. Goal to limit movement from ProjectHydra to graveyard (Path to graveyard should be from Labs)

  2. List sent to Hydra list for approval / confirmation
